# 清居智能：大模型调度引擎高频面试 Q&A

适用范围：AI 基础设施分层、多模型路由、熔断、模板方法、SSE、首包探测、Embedding 和 Rerank。清居智能的订单、库存、物流属于模拟 MCP 数据；模型调度只保证模型调用可靠，不替代业务数据校验。

## 1. 为什么不在 RAG 业务代码里直接写百炼、Ollama 或硅基流动的 HTTP 调用？

**答：** 业务层会被 URL、鉴权、请求格式、错误码和供应商差异污染，新增模型时需要修改多处。项目将能力接口、路由层和供应商 Client 分开：Pipeline 只依赖 Chat/Embedding/Rerank 服务，路由层选模型，Client 处理协议细节。清居智能从云端切到本地 fallback 时不改检索和 MCP 业务代码。

## 2. 模型配置驱动具体解决什么问题？

**答：** 候选模型的 provider、模型名、优先级、能力和端点由配置描述，运行时组装为 `ModelTarget`。这样切换清居智能的默认 Chat 模型、增加本地模型或替换 embedding 模型只改私有配置；代码只保留稳定的选择与调用策略，避免把密钥和模型名散落在业务类中。

## 3. `ModelSelector` 选择候选模型时，只按 priority 排序吗？

**答：** priority 只是基础顺序，还要根据能力筛选，例如深度思考、Chat、Embedding、Rerank 是否支持，并结合熔断健康状态过滤不可用目标。最终候选列表不是硬编码单一模型；若没有可用候选，应清晰失败而不是让业务层空指针。

## 4. 为什么要有 `ModelTarget`，直接传 provider 和 model 字符串不行吗？

**答：** `ModelTarget` 将候选配置、provider 配置和实际模型标识封装为一次调用目标，减少参数散落和重复解析。Client 因而能统一获取 URL、鉴权、模型名和能力；日志、熔断器和 Trace 也能按稳定的 target ID 记录。

## 5. 三态熔断器的 CLOSED、OPEN、HALF_OPEN 各是什么意思？

**答：** CLOSED 正常放行并累计失败；连续失败达到阈值后进入 OPEN，直接拒绝调用以保护下游；冷却时间到后进入 HALF_OPEN，只允许有限探测请求。探测成功回 CLOSED 并清零失败计数，失败重新 OPEN。它保护的是模型供应商故障时的雪崩，不是业务参数错误。

## 6. 为什么 HALF_OPEN 只允许一个探测请求？

**答：** 若放大量请求同时探测，故障服务刚恢复或仍异常时会再次被流量打穿，也会产生大量重复失败。项目用并发安全标记限制探测名额，成功后恢复正常流量；这对清居智能高峰期商品咨询尤其重要，因为同一故障不能让所有请求反复撞向失效模型。

## 7. 同步调用的 fallback 是如何保证不会无限重试的？

**答：** 路由执行器按有序候选列表逐个尝试，每次只调用一次；成功标记健康，失败分类并标记失败，随后切下一个候选，全部失败才返回统一远程异常。候选列表是有限的，熔断 OPEN 的目标会在调用前被跳过，因此不会形成无限循环。

## 8. `AbstractOpenAIStyleChatClient` 用模板方法模式解决了什么？

**答：** 多个供应商大多兼容 OpenAI Chat Completions，通用流程包括校验配置、构造 JSON、发送 HTTP、解析响应和转换异常。基类实现这些稳定步骤，子类只定义 provider 名、是否需要 Key、特有字段或 reasoning 行为；新增供应商的改动集中，符合开闭原则。

## 9. 为什么要区分 HTTP 错误、网络错误和响应格式错误？

**答：** 它们的可恢复性不同：401/403 多半是配置问题，429/5xx 可能适合 fallback，网络超时可重试或切换，非法 JSON 是供应商协议或解析问题。统一包装为 `ModelClientException` 后，路由层能做一致的健康标记和日志，而业务层只处理“本次模型不可用”。

## 10. 流式调用为什么不能复用同步调用的失败判断？

**答：** 同步调用拿到完整响应才能判断；流式调用 `streamChat()` 很快返回句柄，真正 token 在后台线程陆续到达。仅凭 HTTP 建连成功不能判定成功，可能长时间无首包或首包前异常，因此需要专门的首包探测机制。

## 11. `ProbeStreamBridge` 怎么避免切换模型时前端看到重复内容？

**答：** 它位于供应商回调和真实前端回调之间，首包确认前先缓冲 `onContent/onThinking/onComplete/onError` 动作。首包成功才 `commit` 并回放缓存；超时、错误或无内容结束则取消当前 Call 并切换候选，因此失败模型的半截内容不会泄漏到浏览器。

## 12. 首包等待和后台读 SSE 是同一个线程阻塞吗？

**答：** 不是。路由线程在 `CompletableFuture.get(timeout)` 等 `ProbeResult`，模型流线程在 OkHttp `readUtf8Line()` 等供应商 SSE 行；前者收到 bridge 信号后决定是否提交，后者持续读流并回调。区分两个阻塞点，才能正确解释 TTFT、超时和取消的时序。

## 13. 用户取消时，如何真正打断阻塞的模型读流？

**答：** `StreamCancellationHandle` 持有取消逻辑，当前实现设置共享 cancelled 标志并取消 OkHttp Call。取消 Call 会让底层阻塞读取尽快退出；回调层还会检查任务是否已取消，防止少量在路上的 token 继续写入会话或发送 SSE。

## 14. Embedding 客户端为什么也需要路由和容错？

**答：** 入库、重建索引和查询向量都依赖 embedding 服务，单模型不可用会导致资料无法检索或无法入库。Embedding 路由与 Chat 路由复用候选、健康和 fallback 思路，但必须额外校验向量维度；清居智能切换 embedding 模型时需要重建受影响的向量数据。

## 15. Rerank 与向量召回分别做什么，为什么不只保留一个？

**答：** 向量召回从大规模候选中快速找语义相近内容，追求 Recall；Rerank 对少量候选做更精细的 query-document 相关性判断，追求 Precision。清居智能中“滤芯多久换”和“滤芯是否兼容”词很像但证据不同，Rerank 可减少错误条款进入 Prompt；失败时要能退化到召回排序。

## 16. 为什么项目提供 NoopRerankClient？

**答：** 空对象模式让调用链始终有一个可执行实现，避免每处 `if (rerankEnabled)` 分支。开发环境、模型不可用或成本敏感时它原样返回候选；业务仍可回答，且配置和 Trace 能显示当前没有进行真实精排。

## 17. API Key、模型 URL 应该放在哪里？

**答：** 代码库只保存结构和非敏感默认值，真实 Key 放本地私有配置、环境变量或密钥管理系统；日志中不输出 Authorization 头和完整密钥。高德地图等后续外部 MCP 同样通过独立配置绑定 Key，不能硬编码到工具提供者里。

## 18. 你如何观察一次清居智能回答到底慢在模型、检索还是首包？

**答：** Trace 记录改写、意图、检索、路由、首包和供应商流节点的耗时；用户感知指标看 TTFT，而不是只看总耗时。若 TTFT 高且检索节点正常，优先看模型路由、首包探测和供应商健康；若检索高，再检查向量库、重排和并发池，而不是盲目换模型。
