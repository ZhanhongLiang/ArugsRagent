# 清居智能：AI 知识问答高频面试 Q&A

适用范围：RAG Pipeline、记忆、改写、意图树、检索、MCP、Prompt、SSE、限流和评测。问题按严厉实习面试官的追问方式组织；清居智能使用 50 SKU 和模拟订单数据，不把模拟数据包装成真实线上业务。

## 1. 从用户问“我上次看的那款适合卧室吗”开始，完整链路是什么？

**答：** 请求先经过幂等和全局并发控制，再进入流式 Pipeline：加载会话记忆、改写与拆分、意图识别、歧义引导、系统直答判断、检索或 MCP 调用、Prompt 组装、LLM 流式输出。这里“上次看的那款”需要历史补全，“卧室”可能需要房间面积等条件；若型号不明确，系统应澄清而不是猜测。

## 2. 为什么不把所有问题都直接向量检索？

**答：** 不同问题的数据性质不同：商品说明和保修规则适合 RAG；订单、库存、物流状态是实时结构化数据，必须走 MCP；问候语不需要检索。把订单状态入库后再检索会有延迟和陈旧数据风险，也无法做订单号、手机号后四位等参数校验。

## 3. 多轮对话的记忆是怎么持久化和加载的？

**答：** 每轮 user/assistant 消息落到 PostgreSQL，加载时并行读取摘要和最近历史，再按 system 摘要、历史消息、当前问题的顺序注入模型请求。清居智能中，用户先问“VC-A2”，下一轮问“它的滤芯多久换”，改写器可从历史补齐成“VC-A2 的滤芯多久更换”。

## 4. 为什么滑动窗口不够，还要做摘要压缩？

**答：** 只保留最近 N 轮会丢掉早期但关键的型号、预算、家庭场景；全量携带又会导致 Token 成本和延迟持续增长。项目将早期消息压缩为结构化摘要，同时保留近期原文；摘要任务异步执行并带水位线，避免每新增一条消息都重复压缩同一段历史。

## 5. 用户问“这个能除螨吗，还能不能查订单”，为什么要拆问题？

**答：** 前半句是商品能力检索，后半句是实时订单查询，属于不同意图和数据源。改写器一次 LLM 调用同时输出改写后的主问题和子问题；后续每个子问题独立路由，最后由 Prompt 把知识库结论和 MCP 结果合成。

## 6. LLM 改写失败后，系统是不是就不能回答？

**答：** 不能把非关键增强变成单点故障。项目先做术语归一化，再调用 LLM 改写和拆分；失败时回退到归一化后的原问题和规则拆分，保证主链路继续。比如把“吸尘器头”归一为“吸头”，即使改写模型不可用，仍可检索。

## 7. 50 SKU 为什么还需要三级意图树，关键词分类不够吗？

**答：** SKU 少不代表问题类型少。清居智能至少有品类、商品咨询、规格比对、配件兼容、售后政策、订单物流、库存等意图；三级树把领域、类别和叶子能力分开，叶子节点可绑定知识库、TopK、Prompt 或 MCP 工具。关键词规则适合作为补充，无法可靠处理“这个刷头能用在我那台吗”这类口语和指代。

## 8. 一次给几十个意图打分，如何防止模型返回一堆低质量结果？

**答：** 先只把可用叶子节点序列化给模型，并要求受约束 JSON 输出；服务端再做分数阈值过滤、每个子问题 TopN 限制和全局总量封顶。总量封顶不能只按全局分数截断，否则某个子问题可能一个意图都没有；应先保障每个子问题保留最高分，再分配剩余名额。

## 9. 用户问“退货政策”，但没有说是主机、耗材还是延保，系统怎么办？

**答：** 这是歧义，不应同时拼接三套政策再让模型自己猜。项目按品类聚合候选意图、比较头部得分差和置信区间，必要时用 LLM 二次确认；确认模糊后通过同一 SSE 通道返回澄清问题，例如“您咨询的是主机、耗材还是延保服务的退换政策？”并短路后续检索。

## 10. 意图确定后，KB、MCP、SYSTEM 三种类型分别如何处理？

**答：** SYSTEM 直接用系统 Prompt 回答；KB 进入意图定向和全局向量双通道检索；MCP 依据 `mcpToolId` 找工具定义、提取参数并执行。清居智能的“QJ20260713008 到哪了”命中物流工具，“QJ-ACC-002 是否兼容 VC-A2”可以走兼容 MCP 或具备版本管理的兼容知识库，关键是先明确数据的新鲜度要求。

## 11. 为什么要同时做意图定向检索和全局向量检索？

**答：** 定向检索精度高，能将“VC-A2 续航”限制在无线清洁资料；但意图分类可能漏判或资料跨库，全局向量检索提供召回兜底。两个通道并行执行并合并候选，随后去重和重排；并不是把两个结果直接全部塞给模型。

## 12. 多通道并发会不会把线程池打满？

**答：** 项目区分子问题级 `ragContextExecutor` 和通道级 `ragRetrievalExecutor`，避免所有任务共用一个无界线程池；每层都有容量和拒绝策略。并发的前提是任务独立，最终仍在上层汇总 Future；对外部向量库或 MCP 还要设置超时和单通道失败隔离。

## 13. 三个通道各召回 10 条，为什么最后只给模型 5 条？

**答：** 召回阶段追求不漏，生成阶段追求上下文精确和 Token 可控。后处理链先按内容或来源去重，再用 rerank 根据“当前问题—候选块”的相关性排序，最后截取 TopN。清居智能中，产品页、说明书和 FAQ 可能重复描述同一参数，不去重会挤掉真正的售后限制条款。

## 14. Rerank 失败怎么办？是否整条问答失败？

**答：** 不应该。重排是质量增强层，失败时记录告警并回退到去重后的原始排序；项目还提供 NoopRerankClient 保持调用链稳定。这样“可回答”优先于“完全依赖重排才可回答”，但评测报告要能看出重排降级发生的频率和质量影响。

## 15. MCP 工具是如何被发现并执行的？

**答：** Bootstrap 根据 `rag.mcp.servers` 连接 MCP Server、完成协议握手并调用 `listTools()`；每个远端 Tool 会包装成 `McpClientToolExecutor` 注册到 `McpToolRegistry`。运行时意图节点上的 `mcpToolId` 查到执行器，执行器再发送 `CallToolRequest`。清居模块新增工具后无需改注册中心，启动发现即可注册。

## 16. 用户一句“查下订单 13008”，工具需要的完整参数从哪里来？

**答：** 参数提取器把工具的 JSON Schema、意图节点的专属参数 Prompt 和用户问题交给 LLM，要求只输出合法 JSON；随后做 JSON 清洗、类型转换和默认值填充。若订单工具还需要手机号后四位而用户没给，不能猜测，应返回澄清或让工具给出安全的缺参提示。

## 17. 为什么不让 LLM 直接生成工具参数和答案，不需要 Schema？

**答：** Schema 是工具边界：约束字段名、类型、必填项和枚举，避免模型把“订单号”写成无关字段或给库存查询附加危险参数。LLM 负责理解自然语言，工具负责确定性执行；返回数据再由 LLM 做表达，不能反过来让模型伪造业务状态。

## 18. KB 上下文、MCP 数据和历史消息在 Prompt 中怎么摆放？

**答：** system 消息放角色和安全规则，历史维持会话连续性，KB/MCP 结果作为受明确边界约束的上下文，当前问题最后出现。项目区分 KB-only、MCP-only 和 mixed 场景，温度也可不同；实时订单数据要标明来源和查询时刻，要求模型不能编造未返回字段。

## 19. 为什么静态商品问答和 MCP 订单问答的 Temperature 不同？

**答：** 商品规格、兼容关系和售后政策强调准确复述，温度应低；MCP 结果的语言组织也应稳定，但如果混合场景需要解释比较、给出使用建议，可适当放宽。Temperature 不是“越低越好”，而是根据输出允许的创造性和可验证性分档。

## 20. SSE 流式回答从模型到浏览器经过哪些层？

**答：** 模型客户端在线程池阻塞读取供应商 SSE 行，解析 `content/reasoning_content/[DONE]` 后回调；回调经首包探测桥和 Trace 包装后交给 `StreamChatEventHandler`，再由 `SseEmitterSender` 发送给浏览器。前端收到的协议包含 `meta`、增量 `message`、终态 `finish/done`，而非直接透传供应商的原始 SSE。

## 21. 流式模型为什么要做首包探测，连接成功不算成功吗？

**答：** HTTP 建连成功可能一直没有 token，也可能供应商在首包前报错；如果此时把流暴露给用户，后续切换备选模型会造成重复或断流。`ProbeStreamBridge` 先缓存回调，路由线程等待首个有效内容、错误、空完成或超时；确认首包成功后才 commit 并向前端回放缓存内容。

## 22. 用户点击停止生成，集群里具体怎么停？

**答：** 前端携带 `taskId` 调停止接口；任意节点先将取消标记写入 Redis，再通过 Pub/Sub 广播。真正持有任务的节点用 CAS 确保只取消一次，取消 OkHttp Call、阻止后续 token 回调、保存已生成部分，并发送 `cancel -> done`。Redis 标记还覆盖“取消先到、任务后注册”和“句柄后绑定”的竞态。

## 23. 为什么聊天限流不是简单的 QPS 限流？

**答：** LLM 请求耗时长且 Token 成本高，真正需要控制的是同时在跑的生成数。项目用可过期许可控制全局并发，用 ZSET 按序排队，Lua 原子判断队头资格；拿不到许可的请求走定时轮询和 Pub/Sub 唤醒，超时才拒绝。

## 24. 为什么队列用 ZSET，不用 Redis List？

**答：** ZSET 的 score 可以保存全局递增序号，实现跨实例 FIFO，并能按成员删除取消或超时请求；还可以结合存活标记清理僵尸条目。List 对“查排名、按 requestId 移除、处理失活请求”不如 ZSET 直接。

## 25. 清居智能只有模拟数据，你怎么做评测，结果可信么？

**答：** 模拟数据不等于不能评测，但必须明确它衡量的是系统机制，不是线上业务收益。我会构建覆盖 50 SKU 的规格、兼容、售后、订单、拒答和多轮指代样本；结构化订单工具类题单独验证工具参数正确率和返回一致性，RAG 类题验证召回与忠诚度，不能把两类题混成一个分数。

## 26. 自建指标和 RAGAS 分别解决什么问题？

**答：** 自建指标衡量可确定的工程结果，如意图 Top-1、Recall@K、MRR、工具成功率、TTFT 和错误率；RAGAS 用 Judge 模型评估上下文相关性、回答忠诚度和回答相关性等内容质量。两者结合才能定位问题：检索指标差先调检索，检索正常但忠诚度差再查 Prompt、上下文截断或生成模型。

## 27. RAGAS 分数提高了，你如何证明不是 Judge 偏置？

**答：** 不把单次单模型打分当结论。应固定评测集、记录模型和 Prompt 版本，多次运行看方差，必要时更换 Judge 并对关键样本做人审；中文 NaN、成本、同源偏置都要在报告中暴露。最终以 A/B 差异、失败样本和人工复核形成迭代闭环。

## 28. 你最担心清居智能场景里哪类幻觉，如何控制？

**答：** 最危险的是把实时订单状态、库存、售后资格说错，其次是错误的配件兼容结论。前者强制 MCP 查询并携带参数校验，后者以版本化兼容表为准；Prompt 要求没有证据就明确说明未查到，不允许用常识补全。评测时单列“错误承诺”和“错误兼容”高风险样本。
