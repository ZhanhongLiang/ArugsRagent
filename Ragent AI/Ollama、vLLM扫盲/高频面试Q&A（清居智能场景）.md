# 清居智能：Ollama 与本地模型高频面试 Q&A

适用范围：本地部署动机、Ollama 架构、模型管理、OpenAI 兼容接口和本地模型选型。清居智能的本地模型定位是开发调试或云模型故障时的降级能力，不将个人机器性能包装成生产 SLA。

## 1. 既然云端模型效果更好，清居智能为什么还要接 Ollama？

**答：** 本地模型主要解决开发调试、隐私敏感资料不出网、网络受限和云模型故障降级，不是无条件替代云端大模型。清居智能可用本地 embedding 做说明书入库测试，用本地 chat 验证 Pipeline；面对复杂的商品比较和参数抽取，仍要通过评测决定是否使用云端模型。

## 2. Ollama 在系统中是什么角色？它是模型本身吗？

**答：** Ollama 是本地模型运行与管理服务，负责拉取、存储、加载和通过 HTTP API 推理；模型权重是独立的模型文件。Java 服务不直接加载权重，而是像调用远程模型一样调用 Ollama 的服务端接口。

## 3. 为什么项目能用 Ollama，却不需要在业务代码里写一套特殊逻辑？

**答：** `infra-ai` 将 Chat、Embedding、Rerank 抽象成统一接口，供应商差异封装在 Client 和配置中。Ollama 只需实现或复用 OpenAI 风格协议适配，路由层根据候选模型选择它；业务 RAG Pipeline 只依赖 `LLMService`，不关心底层是云端还是本地。

## 4. Chat 模型和 Embedding 模型能否共用一个？

**答：** 不应想当然共用。Chat 模型优化生成和指令遵循，Embedding 模型优化文本向量语义空间；清居智能的商品说明书检索需要稳定维度和相似度表现，问答生成则需要中文理解与流式能力。配置中应分别声明候选、模型名和维度约束。

## 5. 为什么向量模型切换时必须关注维度？

**答：** 向量库 collection 的维度与 embedding 输出维度必须一致，否则不能写入或检索。切换模型不能只改模型名：要验证新旧维度、决定重建 collection 或迁移索引，并重新向量化所有清居商品分块；旧向量和新向量不能混用。

## 6. Modelfile 有什么用，能不能用它解决所有 Prompt 问题？

**答：** Modelfile 可指定基础模型、系统提示、参数和模板，适合统一本地推理默认行为。它不能替代业务 Prompt：清居智能的检索上下文、订单 MCP 结果和当前问题是每次请求动态变化的，仍要由应用层组装；把所有规则写死在模型侧会降低可观测性和可测试性。

## 7. Ollama API 和 OpenAI 兼容 API 为什么要同时存在？

**答：** 原生 API 便于使用 Ollama 特有能力，OpenAI 兼容 API 便于现有客户端、SDK 和多供应商抽象复用。项目优先统一到 OpenAI 风格 Chat Completions，使 `AbstractOpenAIStyleChatClient` 能复用请求构造、SSE 解析和错误处理；是否切换协议取决于兼容性和所需特性。

## 8. 本地模型部署时，显存不足会怎样处理？

**答：** 先根据显存、上下文长度和并发目标选择参数规模与量化等级，不能只看模型名字；显存不足时可能部分层落到 CPU，延迟会明显恶化。实习项目应如实记录本机模型、量化和测试并发，避免声称本地模型能稳定承载生产级清居订单咨询流量。

## 9. 清居智能如何选择本地模型作为降级候选？

**答：** Chat 候选至少验证中文商品问答、工具参数 JSON 输出、流式首包、长上下文和异常恢复；Embedding 候选验证 SKU 别名、型号、配件编码和政策术语的检索质量。候选顺序由优先级和健康状态共同决定，本地模型应作为明确的 fallback，而不是无条件抢占主模型。

## 10. 本地模型会不会让订单、库存数据更“实时”？

**答：** 不会。模型部署位置影响推理成本、网络和隐私，不改变业务数据一致性；订单、库存、物流仍必须调用 MCP 工具的数据源。本地模型只负责理解“查订单 QJ20260713008”的意图和组织工具结果，不能从参数规模推导出订单状态。
