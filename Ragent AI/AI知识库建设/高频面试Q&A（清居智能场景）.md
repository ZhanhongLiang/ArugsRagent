# 清居智能：AI 知识库建设高频面试 Q&A

适用范围：知识库架构、上传、分块、对象存储、并发控制、定时同步与文档管理。清居智能当前是 50 SKU 的模拟电商场景；商品说明、保修政策、配件兼容规则属于知识库，订单、库存、物流状态属于 MCP 工具数据。

## 1. 你只有 50 个 SKU，为什么还要做 RAG 知识库，数据库查询不就够了吗？

**答：** SKU 数量不大，但每个 SKU 会关联说明书、参数表、保修政策、清洁指南、配件兼容表和活动规则，真正检索对象是非结构化长文，不是 50 行商品主数据。结构化字段如 SKU、价格、库存走业务查询或 MCP；“细软发适合什么风嘴”“滤芯多久更换”等语义问题走 RAG。50 SKU 是冷启动规模，架构按可扩展到多品类、多版本说明书设计，但我不会把它说成海量电商数据。

## 2. 清居智能的知识库应该怎么拆？按一个总库还是按品类拆？

**答：** 我会按“无线清洁、空气护理、个护美发、耗材配件、售后政策”建立逻辑知识域，而不是把全部文档塞进一个库。意图命中某个品类后先定向检索，降低不同产品的参数和政策互相污染；全局向量通道保留作兜底。订单、物流、库存不入向量库，因为它们要求实时性和一致性，应通过 MCP 查询。

## 3. 你设计知识库的核心数据模型是什么？

**答：** 核心是知识库、文档、文档分块和向量索引四层：知识库负责业务边界，文档保存来源和处理状态，分块保存可检索正文及元数据，向量库负责近似语义召回。清居智能的分块元数据至少包含 `sku`、`model`、`category`、`documentType`、`version` 和 `enabled`，这样可以做过滤、溯源和灰度启停。

## 4. 商品规格表、说明书和售后政策的分块策略为什么不能一样？

**答：** 规格表的字段关系紧密，型号、参数、单位必须保留在同一语义块；说明书适合按章节或标题做结构感知分块；售后政策按条款和适用范围分块，避免把“七天无理由”和“耗材不支持退换”拆散。统一固定字数切分会破坏字段关系，模型可能把 A2 的续航和 A3 的价格拼成错误答案。

## 5. 上传接口为什么不直接把 MultipartFile 全读入内存再上传对象存储？

**答：** 大文件同时上传时，文件字节、解析缓冲、字符串解码和对象存储 SDK 缓冲会叠加，单个 30MB 文件的实际堆占用可能远大于文件本身。正确做法是限制 multipart 大小，尽可能走临时文件或流式转发；超大文件可用预签名 URL 让浏览器直传对象存储，应用只接收元数据和回调。

## 6. 预签名直传解决了什么问题，代价是什么？

**答：** 它绕过应用服务器传输大文件，降低 Tomcat 线程、网络带宽和 JVM 堆内存压力；浏览器拿到短时、限定对象键和方法的上传凭证后直接上传对象存储。代价是应用不能在上传字节经过时立即深度校验，因此要限制有效期、对象键前缀、Content-Type 和大小，并在回调或采集阶段再次校验文件真实性。

## 7. 文件上传为什么采用并发数限流，不只做 QPS 限流？

**答：** 上传和解析是长耗时、资源占用型任务，真正稀缺的是同时占用的内存、CPU、Tika 解析器和对象存储连接，不是瞬时请求数。QPS 即使很低，慢上传也可能长期堆积；项目用带租约的分布式许可控制全局并发，实例扩容后仍不会突破总容量。

## 8. 为什么限流放在业务层，而不是只放网关？

**答：** 网关适合做 IP、路径、QPS 等通用保护，但它看不到“本次是否要解析 PDF、文件大小、当前知识库任务负载”等业务语义。清居智能中，上传、分块和向量化的资源成本不同，业务层能根据具体操作获取和释放许可；网关可作为第一道粗保护，两层并不冲突。

## 9. 文档上传和“开始分块”为什么拆成两个接口？

**答：** 上传的目标是可靠保存原文件并创建文档记录；分块、解析、增强、向量化属于长耗时异步任务。拆开后前端能先拿到稳定的文档 ID，再让用户确认或重试处理；清居智能可先上传一批说明书，检查分类和版本正确后再触发入库，避免错误资料立即污染检索结果。

## 10. 开始分块时为什么需要事务消息和状态 CAS？

**答：** 需要保证“文档状态变为处理中”和“分块事件可被消费”不会只成功一半。生产者本地事务里用条件更新把状态从非 RUNNING 切到 RUNNING，更新行数为 0 说明已经有任务在处理；事务消息确认后消费者才异步分块。这样重复点击不会产生两个并发分块任务，消息投递失败也不会把文档伪装成已处理。

## 11. 消费者重复收到分块消息怎么办？

**答：** 消息系统通常至少一次投递，不能假设只收到一次。分块任务要以文档状态、任务 ID 或幂等键做防重，写入分块时采用可重复执行的策略；失败时记录节点日志和错误原因，允许从失败状态重试，而不是简单吞异常。

## 12. 定时同步远程商品手册时，如何避免每次都重新向量化？

**答：** 先通过远程文件的 ETag、Last-Modified、内容哈希或版本号判断是否变化，未变化就跳过；变化后才进入抓取、解析、分块和索引流程。清居智能的滤芯说明书若版本没变，不应每天重复生成 embedding；版本变更后应保留新版本并使旧分块失效或下线，避免两版政策同时命中。

## 13. 集群中多个实例都在扫描定时任务，怎么避免重复同步？

**答：** 扫描可以在每个实例执行，但真正领取任务时必须获取分布式锁或租约，只有持锁实例执行；锁需要续租和过期兜底，防止实例崩溃后任务永久卡住。任务表还要有状态机和超时恢复，将长时间 RUNNING 的任务恢复为可重试状态。

## 14. 文档或分块被禁用后，为什么不能只改数据库状态？

**答：** 数据库状态和向量索引必须保持一致。若只把分块标成禁用，向量库仍可能召回它；因此需要同步删除、过滤或更新向量侧元数据，并清理缓存。清居智能下线过期的保修政策时，必须确保旧条款不会继续被模型引用。

## 15. 如何证明知识库资料真的提升了清居智能问答，而不是“上传了很多文档”？

**答：** 先按品类、规格比对、配件兼容、售后政策、口语化问法构建带标准证据的评估集，再观察 Recall@K、上下文准确度、答案忠诚度和无答案拒答率。对例如“VC-A2 能不能用 QJ-ACC-002”这类题，要同时检查检索是否找对兼容表、最终答案是否没有越过文档乱说。
