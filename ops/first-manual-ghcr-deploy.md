# First Manual GHCR Deployment Handoff

Stage 3 only publishes immutable GitHub Container Registry images. It does not SSH to a server, does not run a deployment command, and does not read or write production secrets.

## 1. Identify the release SHA

1. Open the GitHub repository Actions tab.
2. Select the `Ragent CI and GHCR Release` workflow.
3. Use a successful run created by either:
   - a `push` to `master`; or
   - a manual `workflow_dispatch` run from `master`.
4. Copy the full 40-character commit SHA from the workflow run summary.

The deployable tag is exactly that full SHA. Do not use `latest`, `master`, a branch name, a date tag, or a short SHA.

## 2. Immutable image repositories

The workflow publishes exactly one SHA tag for each image:

```text
ghcr.io/zhanhongliang/arugsragent-api:<FULL_40_CHAR_GIT_SHA>
ghcr.io/zhanhongliang/arugsragent-web:<FULL_40_CHAR_GIT_SHA>
```

These repository bases are fixed by the Stage 3 contract:

```text
ghcr.io/zhanhongliang/arugsragent-api
ghcr.io/zhanhongliang/arugsragent-web
```

## 3. Server-only inputs before the first deployment

Before running the first deployment on the server, populate the existing Stage 2 server files with server-only Ragent values according to the installed Stage 2 scripts:

```text
/opt/ragent/shared/registry.env
/opt/ragent/shared/runtime.env
/opt/ragent/shared/application-prod.yaml
/opt/ragent/shared/control.env
```

These files are intentionally not generated by GitHub Actions. They may contain host paths, registry pull credentials, database credentials, middleware endpoints, object storage credentials, model provider keys, and the health-check contract used by Stage 2.

Do not commit those values to Git.

## 4. GHCR authentication on the server

If the GHCR packages remain private, authenticate from the server using a server-held GitHub token with package read permission. The token must stay on the server side only, typically through `/opt/ragent/shared/registry.env` or the Stage 2 login script contract.

Example shape only, with no real token:

```bash
docker login ghcr.io -u <github-user> --password-stdin
```

Never paste the token into this repository, the workflow YAML, or deployment documentation.

## 5. Verify image manifests for the exact SHA

On the server, after authentication if needed, verify both manifests for the copied full SHA:

```bash
docker manifest inspect ghcr.io/zhanhongliang/arugsragent-api:<FULL_40_CHAR_GIT_SHA>
docker manifest inspect ghcr.io/zhanhongliang/arugsragent-web:<FULL_40_CHAR_GIT_SHA>
```

Both commands must succeed before deployment.

## 6. First manual server deployment sequence

Run the Stage 2 server-side flow manually:

1. Validate all server-only inputs with the installed Stage 2 validation script.
2. Deploy the exact full SHA with the installed Stage 2 release command:

   ```bash
   /opt/ragent/bin/deploy-release.sh <FULL_40_CHAR_GIT_SHA>
   ```

3. Run the Stage 2 health checks defined by `/opt/ragent/shared/control.env`.
4. Compare the protected Docker snapshot produced by Stage 2 before and after the deployment.
5. Keep the full SHA recorded as the deployed release identifier.

This document intentionally does not claim a concrete health URL. The authoritative health-check value is the Stage 2 `control.env` contract on the server.

## 7. Scope boundary

Argus is unrelated to this first Ragent deployment handoff. Do not touch Argus services, containers, networks, ports, configuration, deployment scripts, or server ownership while deploying this release.

Automatic SSH deployment is intentionally deferred to a later stage after the first manual GHCR release succeeds and the server-side Stage 2 process is proven.
